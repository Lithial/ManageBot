# wrap Phase 9: Real MCP + E2E smoke

> Base: `main` (Phase 8 merged). The final phase. Split into **3 PRs** given size.
> User picks: real MCP server now; e2e gated on `claude` on PATH + its own auth.

## Architecture

- **`wrap-mcp` stdio bridge** (`cmd/wrap-mcp`): the MCP server `claude` spawns (configured per worker via a
  generated `--mcp-config`). Built on the official `github.com/modelcontextprotocol/go-sdk`. Exposes the five
  `wrap.*` tools and relays each call to the daemon over the Unix socket, scoped by `--worker <id>`. The daemon
  gains no new network surface.
- **Daemon worker-facing endpoints**: `GET /workers/{id}/task`, `GET /workers/{id}/siblings`,
  `POST /workers/{id}/progress|done|blocked`. The POSTs record `worker_progress` / `worker_report_done` /
  `worker_report_blocked` events (distinct from the orchestrator's finalized `worker_done`/`worker_failed`).
- **Orchestrator outcome shift (9b):** a worker's terminal status comes from its recorded MCP calls (events) +
  exit code, not stdout NDJSON ‚Äî "the daemon never parses output for meaning."

## PR breakdown

- **9a (this PR): MCP bridge + daemon worker endpoints (additive, no orchestrator change).**
  1. `store.GetWorker(id)`.
  2. API worker endpoints + `intake` DTOs (`WorkerTaskResponse`, `SiblingTasksResponse`, report bodies).
  3. `client` worker methods (`WorkerTask`/`WorkerSiblings`/`WorkerReportProgress|Done|Blocked`).
  4. `cmd/wrap-mcp` MCP server (5 tools ‚Üí client); tested with an in-memory MCP client.
  5. Makefile `wrap-mcp` target; docs.
- **9b: switch orchestrator to MCP-driven outcomes + command templates.** Build per-worker `--mcp-config` +
  `--append-system-prompt <planner.md|worker.md|merger.md>`; interpret outcomes from `worker_report_*` events;
  extend `fake-claude` to hit the worker endpoints so integration tests keep working.
- **9c: `//go:build e2e` smoke** ‚Äî one real `claude` run against a trivial fixture, skipped unless `claude` is
  on PATH; documented, not in CI.

## Decisions

- Official MCP **go-sdk** (not hand-rolled): I can't test protocol compat against the real `claude` client here,
  so the SDK de-risks it.
- Endpoints record **`worker_report_*`** events; the orchestrator's finalized verdict events are unchanged. 9b
  reads the report events to determine outcome.
- Bridge is **worker-scoped** by a CLI arg, so the daemon's auth/scoping stays trivial (Unix socket + worker id).
