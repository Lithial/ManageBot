# wrap Phase 6: TUI (Bubble Tea, on the existing API)

> Compact plan. Base: `main` (Phase 5 merged). TDD per task, commit per task, push + PR at the end.

**Goal:** A Bubble Tea TUI over the existing daemon API — **no new daemon/orchestrator logic**. A dashboard
lists all runs (phase + pending gate); selecting one drills into a detail view (plan/merge summary, pending
gate) where the user can approve/reject gates with a keypress. Updates are **poll-based** (a `tea.Tick`
refetching the list or the selected run), matching the daemon's poll API. `wrap tui` opens the list; `wrap
attach <run-id>` opens straight into a run's detail view.

**Decisions (spec-driven):**
- Library: **Bubble Tea** (spec mandates it) + lipgloss for layout. New deps added to go.mod.
- Updates: **polling** (no streaming endpoint). Default ~1s tick.
- Testing: **test the Bubble Tea `Model.Update` and the `tea.Cmd`s** (with a fake client); never snapshot
  rendered `View()` output (per spec testing strategy).

## Key design decisions

1. **The TUI talks to a `tui.DaemonClient` interface, not `*client.Client` directly** — `{ListRuns, GetRun,
   Approve, Reject}`. `*client.Client` satisfies it. This makes `Update`/commands testable with a fake.
2. **One `Model`, two modes** (`modeList`, `modeDetail`). Messages: `runsMsg`, `detailMsg`, `resolvedMsg`,
   `errMsg`, `tickMsg`. Commands: `fetchRuns`, `fetchDetail(id)`, `resolve(id, approve|reject)`, `tick`.
   `Update` is pure over (model, msg) → (model, cmd); all I/O is inside commands.
3. **New read endpoint `GET /runs`** (list, newest first) + `store.ListRuns` + `client.ListRuns` +
   `intake.RunSummary`/`ListRunsResponse`. The only daemon-side addition — a pure read, no orchestration.
4. **Approve/reject reuse Phase 5's** `POST /runs/{id}/approve|reject` via the client; the TUI never writes
   phase. After a resolve, the model refetches detail so the UI reflects the daemon's next tick.
5. **`Model` holds no `*tea.Program`**; `tui.Run(client, startRunID)` builds the model and runs the program.
   `startRunID == ""` → list mode; non-empty → detail mode (attach).
